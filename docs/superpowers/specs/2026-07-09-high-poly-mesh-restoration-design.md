# 高品質メッシュ復元 設計書

- 日付: 2026-07-09
- ステータス: 承認済み（brainstormingフェーズ完了）

## 背景・目的

現在Web公開されているアバターモデル（`model.glb`/`model.usdz`）は、顔・胴体・脚にテクスチャの伸びと黒い破損が発生しており、製品として出せる見た目になっていない。

## 調査で判明した原因

`scripts/prepare_for_mixamo.py`が、元の高品質メッシュをMixamoオートリガー向けに過剰にデシメートしていたことが原因。

| | 元の`ririka.fbx` | 現在使用中の`avatar_idle_mixamo.fbx` |
|---|---|---|
| トライアングル数 | 166,702 | 24,954（-85%） |
| メッシュ数 | 14（Bag/Body/body_b/Boots/Bra/Cloth/…） | 1（全結合） |
| ボーン数 | 337 | 33（Mixamo標準リグ） |
| シェイプキー | 多数（Bodyに719個など） | 0（削除済み） |

14個の別々にUV展開されたメッシュを1つに結合した上で85%デシメートした結果、UV境界をまたいだ頂点統合が発生し、テクスチャの伸び・黒破損として現れている。

## 前提・制約

- 元のArmature（337ボーン、`Hips/Spine/Chest/Neck/Head`等の命名）とMixamoのArmature（33ボーン、`mixamorig:`prefix）はボーン階層が異なり、単純なボーン名対応だけではリターゲティングできない。BlenderにRokoko等のリターゲティング用アドオンが未導入のため、リターゲティングは今回のスコープ外とする
- Mixamoアニメーション（idle/wave/dance切り替え）の仕組みは維持する。Mixamoのオートリガーを使い続け、デシメートを可能な限り軽減する方針で対応する
- 以前166K程度のポリゴン数でMixamoのオートリガーがハングした前例があるため、失敗の可能性は残る

## 実装内容

### 1. `prepare_for_mixamo.py`にCLI引数を追加

現在ハードコードされている`DECIMATE_THRESHOLD`（30000）・`DECIMATE_TARGET`（25000）を、任意指定可能なオプション引数にする。引数省略時は現状のデフォルト値のまま動作し、既存の後方互換性を壊さない。

```
Usage: blender --background --python prepare_for_mixamo.py --
       <input_fbx> <output_fbx> [--decimate-threshold N] [--decimate-target N]
```

### 2. デシメート無効化でフル品質版を準備

`ririka.fbx`を対象に、`--decimate-threshold 200000`（166,702トライアングルを超える値）を指定して実行し、フル品質のまま単一メッシュ化・スケルトン除去したFBXを生成する。

### 3. Mixamo再アップロード（手動ステップ）

生成したFBXをユーザーが手動でMixamo（mixamo.com）にアップロードし、オートリギング→idle/wave/danceの3アニメーションを再取得、`ar-avatar-demo/source/`内の既存3ファイル（`avatar_idle_mixamo.fbx`等）を置き換える。

### 4. 再変換・検証

既存の`convert_to_web.py`（変更不要）で`model.glb`/`model.usdz`を再生成する。プレビューでスクリーンショットを撮り、以前の黒破損が解消されていることを確認する。ファイルサイズを確認し、モバイルで許容範囲か判断する。

### 5. 失敗時のフォールバック

もしMixamoがフルポリゴンでハング・失敗した場合、`--decimate-threshold`/`--decimate-target`を軽めの値（目安: 80,000〜100,000トライアングル。現状の25,000より大幅な改善）に変更してステップ2〜4を再試行する。

### 6. 両プロジェクトへ展開

検証OK後、既存パターンに沿って`ar-avatar-demo`をpush、`vrc-mascot-pwa`にモデルをコピーしてSWキャッシュバージョンを更新・push。

### 7. 実機再確認（手動）

iPhoneで見た目・パフォーマンス（読み込み速度・操作の滑らかさ）を確認する。

## 動作確認方法

1. `prepare_for_mixamo.py`にCLI引数を追加したことをスモークテストで確認（デフォルト引数での既存動作が壊れていないこと）
2. フル品質版FBX生成→ユーザーによるMixamo手動アップロード→再変換→スクリーンショット比較で黒破損が解消されていることを確認
3. ファイルサイズ・実機パフォーマンスを確認し、問題があれば軽めのデシメートで再試行
4. 両プロジェクトのiPhone実機で最終確認

## スコープ外（今回やらないこと）

- Mixamoアニメーションを元の337ボーンArmatureへリターゲティングする対応（アドオン未導入のため）
- シェイプキー（表情等）をWeb表示に持ち込む対応（VRChatでのフェイストラッキング用であり、静止表示には不要）
